一、 核心准则 & 哲学
语言习惯：始终使用 中文 交流。

最高禁令：永远不要执行 git clean（严禁任何参数，尤其是 -fdx）。

开发哲学：

No Spec, No Code：无设计，不编码。

No Approval, No Execute：无授权，不执行。

Spec is Truth：设计文档即唯一真理。

二、 技能 (Skill) 使用约定
根据任务复杂度自动选择模式：

sdd-riper-one (默认)：标准模式，用于常规任务。

sdd-riper-one-light：轻量模式，仅限任务简单、边界清晰、改动极小且无需完整流转时使用。

原生模式：极简单、无需 Spec/Checkpoint/分阶段控制的任务，可不调用 Skill。

三、 执行规则 (Execution Rules)
1. 方案确认流程
代码修改：必须先提交方案并 等待确认 后执行。

文档修改：可直接执行并告知。

收敛原则：在等待确认期间，将方案收敛至 最小改动集，严禁无限制扩大搜索或改动范围。

2. 代码修改策略
增量修改：禁止一次性重写整个文件；优先采取小步快跑、分段、少量多次的修改方式。

拒绝重构：除非明确要求，否则禁止大范围重构、整文件替换或发布超大 Patch。

核心先行：多文件改动时，优先完成核心链路的最小可用修改，再逐步补齐边缘逻辑。

专注任务：严禁“顺手”修改与当前任务无关的代码。

3. 探索停顿点
如果连续 2 轮 以上仅进行阅读/搜索而无实际修改，必须停止探索，并向用户明确说明：

准备修改哪些文件？

先改哪一处？

为什么这么改？

四、 命令与 Git 边界 (Boundaries)
1. 高开销操作
禁止主动执行：编译、打包、测试、部署、迁移。

禁止主动管理依赖：安装、升级、删除依赖或修改锁文件（如 package-lock.json, go.sum 等）。

二次确认：任何不可逆、批量删除、覆盖、重置或外部写操作，必须等待明确授权。

2. Git 规范
尊重配置：严格遵守 .gitignore，严禁主动使用 git add -f。

精确提交：默认不提交被忽略的内容；仅在路径被明确点名时，才允许精确提交。

五、 设计同步 (Spec Sync)
强制同步：当改动影响 需求、接口、行为、约束、流程或实现决策 时，执行后必须立即同步更新 Spec 文档。

豁免条件：仅纯机械性改动（如修复错别字、格式调整）可不更新 Spec。
